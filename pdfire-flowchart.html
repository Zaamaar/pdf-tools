Full Project Timeline
Feb 23, 2026
🚀 Project Kickoff — Architecture Design
Decided on a fully serverless stack: S3 + Lambda + API Gateway. Chose AWS SAM for infrastructure-as-code. Planned the pre-signed URL upload flow to bypass Lambda's 6MB request limit.
✓ Architecture decided ✓ SAM chosen
Feb 23 – 24, 2026
🏗️ Backend Infrastructure — S3, API Gateway, Lambda
Wrote template.yaml defining the S3 bucket (with CORS + lifecycle rules), API Gateway, and the first 4 Lambda functions: merge, split, compress, rotate. First successful sam deploy.
✓ S3 bucket created ✓ API Gateway live ✓ 4 Lambda functions deployed
Feb 24 – 26, 2026
🎨 Frontend — Static HTML/CSS/JS App
Built the entire frontend as a single index.html with no frameworks. Implemented drag-and-drop, file list, progress animations, and tool routing. Hosted on S3 static website. Deployed to pdfire-tools bucket.
✓ Frontend live on S3 ✓ Merge, Split, Compress, Rotate working
Feb 26, 2026
🐛 Issue #1 — JPG↔PDF Routes Never Created
Tested JPG to PDF and PDF to JPG — both returned NetworkError. Curl-tested the API and got {"message":"Not Found"}. Checked API Gateway routes — /jpg2pdf and /pdf2jpg were completely missing.
✗ Routes missing from template.yaml → Added function blocks + redeployed
Feb 26, 2026
🐛 Issue #2 — Wrong Code in jpg2pdf Folder
After deploying, CloudWatch showed: index.handler is undefined or not exported and INFO: altschool assignment, question 2. The folder contained old unrelated assignment code.
✗ Old code deployed instead of new function → Overwrote file with cat > and redeployed
Feb 26 – 28, 2026
🐛 Issue #3 — MIME Type Bug (Drag & Drop Broken)
JPG to PDF still failed after routes were fixed. Root cause: file.type returns empty string on drag-and-drop in browsers. The fallback || 'application/pdf' meant every dragged JPG was sent to S3 as a PDF, crashing the Lambda.
✗ file.type empty on drag-and-drop → Built getMimeType() using file extension as fallback
Feb 28 – Mar 1, 2026
✅ PDF to Word + Word to PDF — New Features Added
Built two new Lambda functions: pdf2word (using pdf-parse + docx) and word2pdf (using mammoth + pdf-lib). Created function folders, package.json files, ran npm install, and added to template.yaml.
✓ pdf2word function built ✓ word2pdf function built
Mar 1, 2026
🐛 Issue #4 — template.yaml Outputs Placement Bug
Used cat >> to append new functions to template.yaml but they landed after the Outputs: block. CloudFormation threw: Template format error: Invalid outputs property: [Type, Properties].
✗ Functions appended outside Resources block → Reconstructed template using head -n + clean rewrite
Mar 1, 2026
🐛 Issue #5 — Word to PDF Unicode Encoding Error
Word to PDF returned Processing Failed. CloudWatch showed: WinAnsi cannot encode "" (0xf4de). Standard PDF fonts only support WinAnsi — smart quotes, em dashes, and special characters in Word docs caused crashes.
✗ Special Unicode chars unsupported by pdf-lib fonts → Added sanitizeText() to strip/replace special chars before render
Mar 1 – 3, 2026
📱 Responsive Design — All Screen Sizes
Added comprehensive CSS media queries for tablet (≤900px), mobile (≤600px), and small mobile (≤380px). Fixed stacking layouts, touch-friendly buttons, full-width download buttons, and collapsing footer.
✓ Fully responsive across all devices
Mar 3, 2026
🐛 Issue #6 — IAM Policy Limit Hit
Tried to add CloudFrontFullAccess to enable HTTPS. Got: LimitExceeded: Cannot exceed quota for PoliciesPerUser: 10. Had 10 policies attached including AmazonEC2FullAccess which wasn't needed.
✗ AWS 10-policy limit reached → Detached unused EC2 policy, attached CloudFront
Mar 4, 2026 — Ongoing
🔒 Issue #7 — CloudFront Account Verification Pending
After fixing IAM policy limit, CloudFront creation failed: AccessDenied: Your account must be verified before you can add new CloudFront resources. AWS Support case opened. Site currently runs on HTTP.
⏳ AWS Support case opened ⏳ HTTPS pending verification

All Issues — Root Causes & Fixes
#1
JPG↔PDF Routes Never Created
curl → {"message":"Not Found"}
API Gateway had no /jpg2pdf or /pdf2jpg routes
Fix: Added Jpg2PdfFunction and Pdf2JpgFunction blocks to template.yaml with correct HttpApi events, then redeployed.
✓ Resolved
#2
Wrong Code in jpg2pdf Folder
Runtime.HandlerNotFound: index.handler undefined
LOG: "altschool assignment, question 2"
Fix: The folder had old assignment code. Overwrote with cat > index.js, pasted correct handler, ran npm install, redeployed.
✓ Resolved
#3
MIME Type Bug — Drag & Drop Broken
file.type = "" on drag-and-drop
Fallback sent every JPG as 'application/pdf'
Fix: Built getMimeType() helper that reads the file extension when file.type is empty or generic. Applied to both the Lambda request and the S3 PUT header.
✓ Resolved
#4
template.yaml Outputs Placement Bug
Template format error: Invalid outputs property
Functions appended after Outputs: block
Fix: Used head -n 298 to cut the file cleanly before Outputs, appended functions in the correct Resources block, re-added Outputs section at the end.
✓ Resolved
#5
Word to PDF Unicode Encoding Error
WinAnsi cannot encode "" (0xf4de)
Standard fonts can't handle smart quotes/em dashes
Fix: Added sanitizeText() to replace smart quotes → straight quotes, em dashes → --, ellipsis → ..., and any remaining non-latin chars → ?. Wrapped drawText in try/catch.
✓ Resolved
#6
IAM Policy Limit (10 max per user)
LimitExceeded: Cannot exceed quota for PoliciesPerUser: 10
Fix: Audited existing policies. Found AmazonEC2FullAccess and AmazonVPCFullAccess were attached with no reason. Detached EC2 policy to free a slot, then attached CloudFrontFullAccess.
✓ Resolved
#7
CloudFront Account Verification Block
AccessDenied: Your account must be verified before you can add new CloudFront resources
Status: AWS Support case opened requesting CloudFront access. This is a standard one-time verification for newer AWS accounts. Site runs on HTTP in the meantime.
⏳ Pending AWS verification
